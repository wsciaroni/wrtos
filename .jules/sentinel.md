## 2024-05-19 - Code Injection via Codegen Configuration
**Vulnerability:** The `wrtos-codegen` tool translates the `schedule.toml` configuration into a static Rust code file (`schedule.rs`). The `task.func` string was being directly formatted into the generated code without any validation. An attacker who controls `schedule.toml` could inject arbitrary Rust code into the generated application by supplying a crafted payload for `task.func` (e.g., `dummy_func], &[], //`).
**Learning:** Build-time configuration files used for code generation must be treated as untrusted input. Directly interpolating strings from configuration files into source code without sanitization or validation creates a code injection vulnerability during the build process.
**Prevention:** Implement strict input validation for all configuration strings used in code generation. Ensure that identifiers and paths conform to the expected language syntax (e.g., allowing only alphanumeric characters, underscores, and colons for Rust paths).
## 2025-02-20 - Memory Exhaustion DoS via Unbounded Configuration
**Vulnerability:** The `wrtos-codegen` tool processes the `schedule.toml` configuration and allocates strings in a loop bounded by the `major_cycle_frames` variable without imposing a maximum limit. A crafted `schedule.toml` with an extremely large `major_cycle_frames` value leads to indefinite memory allocation during the `build.rs` execution, resulting in an Out-Of-Memory (OOM) panic and a Denial of Service (DoS) for the build process.
**Learning:** Any user-provided configuration parameter that dictates resource allocation (such as loop iterations or collection sizes) must be bounded. Unbounded configurations are a direct vector for memory exhaustion attacks, particularly in build scripts or code generators.
**Prevention:** Implement strict upper bound validation for all configuration parameters that drive loops or allocations, returning a clear validation error if the limit is exceeded.
