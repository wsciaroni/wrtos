## 2024-06-25 - Avoid Modulo Arithmetic on Microcontrollers without Hardware Division
**Learning:** In the `wrtos-core` cyclic executive, the frame index was previously advanced using a modulo operation: `self.current_frame = (self.current_frame + 1) % self.major_cycle_frames;`. On target architectures like AVR (e.g., used in `uno-blinky`) that lack hardware division support, the modulo operator translates to a multi-cycle software division routine which is relatively slow. By changing this to a simple add and conditional branch `if self.current_frame >= self.major_cycle_frames { self.current_frame = 0; }`, the same logic avoids a computationally expensive operation, which is critical for maintaining high performance and strict determinism in a real-time system.
**Action:** Always prefer conditional branching (e.g., `if var >= max { var = 0; }`) over modulo arithmetic (`var = (var + 1) % max`) for wrap-around counters in code that targets microcontrollers, particularly those without hardware division instructions, to minimize execution overhead.
## 2024-06-25 - Avoid O(N^2) Nested Loops in Schedule Analysis
**Learning:** In the `wrtos-wcet` crate's `analyze_schedule` function, iterating over every frame and then checking every task to see if it belonged in that frame resulted in an O(F * T * F_T) algorithm. For complex schedules, this loop architecture was a significant performance bottleneck during WCET analysis. By reversing the traversal—iterating over tasks once and accumulating their WCETs into a pre-allocated array of frames—the time complexity drops to O(T * F_T), yielding orders of magnitude faster execution time.
**Action:** When validating or aggregating properties (like execution times) across mappings (e.g., tasks to frames), iterate over the sparse elements (tasks) and map them into dense structures (frame buckets) rather than iterating over every dense bucket and performing sparse checks.
