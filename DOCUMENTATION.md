# wrtos User Guide & Technical Documentation

This document provides a detailed guide on the core concepts, crate APIs, target hardware porting, and safety/fault handling policies of the `wrtos` cyclic executive real-time operating system.

---

## 1. Core Concepts

### Time-Triggered / Cyclic Executive
Unlike event-driven RTOS architectures that rely on priority-based preemptive scheduling and interrupt service routines (ISRs), `wrtos` executes a static, pre-calculated sequence of tasks.
*   **Minor Cycle (Frame)**: A fundamental time slice governed by a periodic hardware timer tick. All tasks scheduled in a frame run sequentially.
*   **Major Cycle**: A repeating timeline containing a fixed number of minor cycles. After the last frame executes, the schedule wraps back to frame 0.

```
Timeline:
|--------- Major Cycle (e.g. 500 ms) ---------|
| Frame 0 | Frame 1 | Frame 2 | Frame 3 | Frame 4 |
|  100ms  |  100ms  |  100ms  |  100ms  |  100ms  |
| Task A  | Task B  | Task A  | Task B  | Task A  |
|         |         |         |         | Task C  |
```

### Determinism Over All Else
*   **No Preemption**: Tasks run to completion on a single shared stack. There is no context switching overhead or stack fragmentation.
*   **No Interrupts**: By disabling or not utilizing interrupts for task triggering, execution times become completely predictable, and race conditions are structurally eliminated.
*   **Frame Overrun Detection**: If the sum of task runtimes in a frame exceeds the frame duration budget, a critical failure is flagged immediately at the frame boundary.

## 2. Crate API & Usage

### 2.1 `wrtos-core`
The core runtime provides the execution loop. To run the operating system, you need:
1.  A struct implementing the `wrtos_core::Timer` trait.
2.  A static schedule array generated by `wrtos-codegen`.
3.  An instance of `wrtos_core::CyclicExecutive`.

#### The `Timer` Trait
To port `wrtos` to any platform, implement this trait:
```rust
pub trait Timer {
    /// Blocks the CPU (e.g., using a hardware timer compare register or low-power sleep)
    /// until the next minor cycle tick occurs.
    fn wait_next_tick(&mut self);

    /// Checks if the previous frame overrun its duration budget.
    fn has_overrun(&self) -> bool;
}
```

---

### 2.2 `wrtos-codegen`
Add `wrtos-codegen` as a build dependency in your application's `Cargo.toml`:

```toml
[build-dependencies]
wrtos-codegen = { path = "path/to/wrtos-codegen" }
```

In your `build.rs` script, read the configuration and generate the schedule:

```rust
// build.rs
fn main() {
    println!("cargo:rerun-if-changed=schedule.toml");
    let toml = std::fs::read_to_string("schedule.toml").unwrap();
    let config = wrtos_codegen::parse_config(&toml).unwrap();
    wrtos_codegen::validate_config(&config).unwrap();
    
    let code = wrtos_codegen::generate_schedule_code(&config);
    let out_dir = std::env::var("OUT_DIR").unwrap();
    let dest = std::path::Path::new(&out_dir).join("schedule.rs");
    std::fs::write(dest, code).unwrap();
}
```

Include the generated schedule in your main source code:
```rust
// main.rs
include!(concat!(env!("OUT_DIR"), "/schedule.rs"));
```

---

### 2.3 `wrtos-wcet`
Integrate the WCET analyzer into your test suite or CI pipeline to verify schedule feasibility:

```rust
#[test]
fn verify_schedule_feasibility() {
    let toml = include_str!("../schedule.toml");
    let config = wrtos_codegen::parse_config(toml).unwrap();

    // Map your task functions to their measured Worst-Case Execution Times
    let mut task_wcets = std::collections::HashMap::new();
    task_wcets.insert("sensor_read".to_string(), std::time::Duration::from_micros(200));
    task_wcets.insert("control_loop".to_string(), std::time::Duration::from_micros(450));

    let report = wrtos_wcet::analyze_schedule(&config, &task_wcets).unwrap();
    assert!(!report.has_overruns, "Schedule contains frame overruns!");
}
```

## 3. Porting to Bare-Metal (ARM Cortex-M Example)

To use `wrtos` on an actual microcontroller (such as an STM32 or nRF52):

1.  **Configure a Hardware Timer**: Configure a hardware timer (e.g., `TIM2`) to update at your desired tick rate (e.g., 1000 Hz).
2.  **Implement `Timer` for the Hardware**:
    ```rust
    struct HardwareTimer {
        timer_register: &'static mut RegisterBlock,
        overrun: bool,
    }

    impl wrtos_core::Timer for HardwareTimer {
        fn wait_next_tick(&mut self) {
            // 1. Wait for update event flag in hardware register (polling)
            while !self.timer_register.sr.read().uif().bit_is_set() {}
            
            // 2. Clear update event flag
            self.timer_register.sr.write(|w| w.uif().clear_bit());

            // 3. Optional: check for overrun flag if timer ticked again before we finished
            if self.timer_register.sr.read().uif().bit_is_set() {
                self.overrun = true;
            }
        }

        fn has_overrun(&self) -> bool {
            self.overrun
        }
    }
    ```
3.  **Run the Cyclic Executive**:
    ```rust
    #![no_std]
    #![no_main]

    use cortex_m_rt::entry;

    mod tasks {
        pub fn sensor_read() { /* Access hardware registers */ }
        pub fn control_loop() { /* Math and actuator write */ }
    }

    include!(concat!(env!("OUT_DIR"), "/schedule.rs"));

    #[entry]
    fn main() -> ! {
        let mut hw_timer = HardwareTimer::new();
        let mut exec = wrtos_core::CyclicExecutive::new(SCHEDULE);
        
        // Starts the infinite execution loop
        exec.run_loop(&mut hw_timer);
    }
    ```

## 4. Safety & Fault Handling

### Handling Overruns
In a safety-critical system, a frame overrun means the system did not complete its actions in the allocated time slot, invalidating the system's deterministic guarantees.
*   **Panic Policy**: The default `CyclicExecutive::run_loop` triggers a Rust `panic!` upon detecting an overrun. On bare-metal, this should map to a hardware watchdog reset or safe shutdown state.
*   **Custom Overrun Logic**: If you prefer custom logging or graceful degradation instead of a panic, use `CyclicExecutive::step` manually inside a custom loop:
    ```rust
    loop {
        if let Err(OverrunError) = exec.step(&mut timer) {
            // Graceful degradation: enter safe state, open relief valves, disable actuators
            enter_safe_mode();
        }
    }
    ```
